# CLAUDE.md — операционная инструкция для Claude Code

Ты разрабатываешь Telegram-бота **с нуля** (greenfield). Это посадочный/вороночный бот под трафик: кандидат заходит по ссылке → инвайт на ближайший созвон + вакансия → пуши → после созвона шлёт триггер-слово → получает контакт менеджера. Полная спека — в `TZ_landing_bot.md` (v0.3), читай её первой, это источник правды.

## 0. Режим работы: с нуля

- Это **новый проект в пустой папке**. Ты создаёшь все файлы сам.
- Рядом могут лежать файлы от прошлого прототипа (HR-бот: `db.py`, `scheduler.py`, `config.py`, `hrbot.service`, `deploy.sh`, `env.example`). **Не импортируй и не редактируй их.** Можешь подсмотреть как референс по стеку (aiogram 3, aiosqlite, APScheduler, systemd-деплой rsync'ом) — но весь код пишешь заново по спеке. Ничего из старого кода не должно оставаться в новом проекте.
- Никаких секретов в коде с самого начала: токен/ID только через env/.env. В прошлом прототипе токен был зашит в `config.py` — не повторяй, не копируй это значение никуда.
- Делай ровно MVP по спеке, без лишней архитектуры: без ORM, без веб-админки, без внешних API созвонов, без Postgres (объём — десятки–сотни лидов/сутки, SQLite достаточно). Все тексты на русском, время везде МСК (`Europe/Moscow`).

## 2. Зафиксированные решения (не переспрашивать)

1. Присутствие/интерес = кандидат прислал **триггер-слово** после созвона. Контакт **не** пушится автоматически.
2. Вакансия — один глобальный текст (в `settings`). В `texts.py` оставь понятный плейсхолдер.
3. Менеджер — один контакт (в `settings`). Ротация — позже, не сейчас.
4. Лимит приглашений на кандидата — **3 суммарно** (1 первичный + 2 реинвайта), один инвайт на созвон. Число — в `settings` (`max_invites`, дефолт 3).
5. Реинвайт получают и `declined`, и «молчуны» (увидели инвайт, кнопку не нажали) — в рамках лимита 3.
6. Защита триггера — **A+C**: триггер-слово задаётся HR на каждый созвон (дефолт неочевидный, не «+»), активно с момента старта созвона до создания следующего созвона. Гейт «только confirmed» (B) **не делаем** в MVP.
7. Source трафика — из deep-link `t.me/<bot>?start=<source>`, пишется в `users.source` только при первом входе, не перезатирается.

## 3. Модель данных (SQLite)

```
users(user_id PK, username, full_name, source, entered_at)
calls(id PK, call_dt, link, trigger_word, status[scheduled|done|canceled], created_at)
attendance(id PK, user_id FK, call_id FK,
           status[invited|confirmed|declined|contact_issued],
           invited_at, rsvp_at, contact_issued_at,
           UNIQUE(user_id, call_id))
settings(key PK, value)   -- vacancy_text, manager_contact, max_invites
```

- `invite_count` кандидата = `COUNT(*) FROM attendance WHERE user_id=?`. При `>= max_invites` в новые созвоны не зовём.
- «Ближайший созвон» = `scheduled` с минимальным будущим `call_dt`. «Только что прошедший» (для триггера) = `scheduled`/`done` с `call_dt <= now`, самый свежий.
- `call_dt` хранить как ISO, tz-aware МСК.

## 4. Порядок сборки

**T0 — скелет.** Структура из §1, `requirements.txt`, `config.py` (env-only, без секретов), `.env.example`, `landingbot.service`, `deploy.sh`, заготовка `README.md`. Локально поднимается пустой бот, отвечает на `/start` заглушкой.

**T1 — db.py.** Схема §3 + async-функции: `init_db`, `upsert_user` (source только при первом входе), `get_setting/set_setting`, `create_call`, `get_next_call`, `get_last_started_call`, `set_call_status`, attendance upsert/get/`set_attendance_status`, выборки для рассылок (кого инвайтить на новый созвон с учётом лимита; кому пуши = `confirmed` по созвону), `invite_count`, агрегаты статистики (воронка общая / по созвону / по source).

**T2 — scheduler.py.** Джобы пушей с id `ping_{kind}_{call_id}`. Пуши 24ч/1ч/10м, на 10м — ссылка, только `confirmed`. Прошедшие отсечки пропускать. `restore_on_startup` перепланирует пуши по **всем** будущим `scheduled` созвонам. Рассылка с троттлингом: батчи + `asyncio.sleep`, ловить `TelegramRetryAfter` (ждать `retry_after`), ловить блокировку юзера и пропускать.

**T3 — keyboards.py + texts.py.** RSVP-клавиатура (`Подтверждаю участие` / `Не смогу в этот день`), админ-меню, тексты инвайта/пушей/выдачи контакта/нейтрального ответа на неверный триггер.

**T4 — bot.py: кандидат.** Точка входа: `init_db()`, `restore_on_startup()`, `AsyncIOScheduler`, polling. Хэндлеры:
- `/start` с парсингом payload → upsert_user(source), привязка к ближайшему созвону (attendance=`invited`), показать вакансию + дата/время + RSVP-кнопки. Нет созвонов — вакансия + «дата уточняется».
- callback `rsvp:confirmed`→`confirmed`, `rsvp:declined`→`declined`.
- обычный текст от кандидата → сверка с триггер-словом последнего стартовавшего созвона в окне активности; верное → `contact_issued` + выдать `manager_contact`; иначе нейтральный ответ.

**T5 — bot.py: /admin (FSM, только `ADMIN_IDS`).** «Назначить созвон» (дата/время + ссылка + триггер-слово → `create_call` + рассылка инвайтов новым и declined/молчунам в рамках лимита + reschedule пушей), «Вакансия» (правка `vacancy_text`), «Контакт менеджера» (правка `manager_contact`), «Статистика» (воронка общая + по последнему созвону + по source), «Экспорт» (CSV кандидатов: статус + source), «Отменить созвон» (cancel + снять пуши).

**T6 — README.md.** Локальный запуск, деплой (systemd), как пользоваться.

## 5. Конвенции

- aiogram 3: Router'ы, `async/await`, type hints. Дату ввода парсить терпимо (`ДД.ММ.ГГГГ ЧЧ:ММ`).
- Идемпотентность: повторный `/start` не плодит юзеров и не трёт source; повторный RSVP обновляет статус.
- Секреты — только в `.env` (есть `.env.example`). В git ничего боевого.
- На каждой задаче T* сначала покажи план/диффы, дождись «го», потом пиши. Коммить по задачам.

## 6. Definition of Done (само-проверка)

- Бот стартует локально с `BOT_TOKEN` из env; `/start?start=src` пишет source.
- HR через `/admin` ставит созвон → подтвердившим приходят пуши 24ч/1ч/10м (актуальные), на 10м — ссылка.
- «Не смогу» → при следующем созвоне приходит реинвайт; после 3 инвайтов суммарно — тишина. Молчуны тоже реинвайтятся.
- После старта созвона верное триггер-слово выдаёт контакт менеджера, неверное/раннее — нет.
- `/admin` → «Статистика» показывает воронку и срез по source; «Экспорт» отдаёт CSV.
- После рестарта процесса пуши по будущим созвонам восстанавливаются.
- В коде и git нет боевых токенов.
