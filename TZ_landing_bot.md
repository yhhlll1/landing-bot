# ТЗ: Посадочный бот (воронка кандидат → созвон → контакт менеджера)

Версия 0.3. База — существующий HR Call Bot (aiogram 3 / aiosqlite / APScheduler / systemd), расширяем его.

---

## 1. Цель

Бот-посадочная для холодного трафика. Кандидат переходит по ссылке, получает приглашение на ближайший общий созвон + описание вакансии, прогревается пушами, доходит до созвона. На созвоне звучит триггер-слово; кандидат, кому интересно продолжить, шлёт его боту и получает контакт менеджера. Цель — максимизировать сквозную конверсию «зашёл → подтвердил → написал триггер → получил контакт» и видеть воронку по источникам трафика.

## 2. Роли

- **Кандидат** — конечный пользователь с трафика.
- **HR / Админ** — назначает созвоны (дата/ссылка/триггер-слово), ведёт вакансию, смотрит статистику (`ADMIN_IDS`).
- **Менеджер** — ботом не пользуется, его контакт выдаётся тем, кто прислал триггер. Пока **один**, позже — ротация.

## 3. Воронка (статусы кандидата по конкретному созвону)

```
entered         — нажал /start (лид зафиксирован)
invited          — привязан к ближайшему созвону, увидел инвайт + вакансию
confirmed        — нажал «Подтверждаю»
declined         — нажал «Не смогу» → ждёт переноса на следующий созвон
contact_issued   — после созвона прислал триггер-слово → выдан контакт менеджера (терминальный, «горячий» лид)
exhausted        — исчерпан лимит приглашений (3), больше не зовём
```

> **Решение №9 (финал воронки):** контакт НЕ пушится автоматически. На созвоне ведущий называет триггер-слово; кандидат шлёт его боту → бот показывает контакт менеджера. Так контакт получают только реально заинтересованные → выше качество лида для менеджера.

Истинная воронка считается из `attendance` (1 кандидат × 1 созвон).

## 4. Сценарий кандидата

1. Переход по `t.me/<bot>?start=<source>` → `/start`.
   - Фиксируем `user_id`, `username`, `source` (payload), `entered_at`. **Source пишется только при первом входе.**
   - Привязываем к ближайшему запланированному созвону → `attendance`, статус `invited`.
   - Показываем: **описание вакансии** (общий текст) + дата/время (МСК) + кнопки `[Подтверждаю участие]` `[Не смогу в этот день]`.
   - Нет активных созвонов — вакансия + «дата уточняется».
2. `[Подтверждаю]` → `confirmed`, идут пуши.
3. `[Не смогу]` → `declined`, открепляем; при следующем созвоне — автоинвайт (лимит §4.7).
4. Пуши (только `confirmed`): за 24ч / 1ч / 10 мин; на 10 мин — **ссылка**. Прошедшие отсечки пропускаем.
5. **Выдача контакта по триггеру.** На созвоне звучит триггер-слово созвона. Кандидат шлёт его в бот:
   - Слово верное И активно (окно) → показываем контакт менеджера → `contact_issued`. Текст: «Отлично! По дальнейшим шагам напишите менеджеру <контакт>».
   - Слово до старта созвона / неактивно / неверное → нейтральный ответ, контакт не выдаём.
6. **Защита триггера (решение по §10):** триггер-слово задаётся на каждый созвон (дефолт — неочевидное, не `+`), активно с момента старта созвона до создания следующего созвона. *(уточняется: добавить ли гейт «только confirmed»)*.
7. **Лимит приглашений:** максимум **3 инвайта на кандидата** (1 + 2 реинвайта). Реинвайт получают `declined` и молчуны (без RSVP). После исчерпания → `exhausted`. Не более одного инвайта на созвон. Число — в настройках.

## 5. Сценарий HR / Админ (`/admin`)

- **Назначить созвон**: дата/время (МСК) + ссылка (руками) + **триггер-слово** (или сгенерировать). Создаёт созвон, рассылает инвайты новым + `declined`/молчунам (в рамках лимита 3).
- **Редактировать вакансию**: один глобальный текст.
- **Редактировать контакт менеджера**: один контакт.
- **Статистика**: воронка общая, по созвону и **по источникам**.
- **Список/экспорт** кандидатов по статусам (CSV).
- **Отменить созвон**: снимает пуши, помечает `canceled`.

## 6. Трекинг источников (deep-link)

Каждой связке/крео — свой `?start=<source>`. Source пишется при первом входе. Вся воронка режется по source → видно, какой источник даёт лучший `entered → contact_issued`, а не просто дешёвые клики.

## 7. Метрики

- Лиды (`entered`) всего и по source.
- По созвону: invited / confirmed / declined / contact_issued.
- Конверсии: entered→confirmed, confirmed→contact_issued (доля заинтересованных из дошедших), сквозная **entered→contact_issued** (главная, по source).
- Реинвайты: возврат `declined`/молчунов в `confirmed`.

## 8. Модель данных (SQLite)

```
users
  user_id PK, username, full_name,
  source TEXT,            -- пишется один раз
  entered_at TEXT

calls
  id PK, call_dt TEXT, link TEXT,
  trigger_word TEXT,      -- слово для выдачи контакта, задаёт HR
  status TEXT,            -- scheduled / done / canceled
  created_at TEXT

attendance               -- 1 кандидат × 1 созвон
  id PK, user_id FK, call_id FK,
  status TEXT,            -- invited/confirmed/declined/contact_issued
  invited_at TEXT, rsvp_at TEXT, contact_issued_at TEXT,
  UNIQUE(user_id, call_id)

settings
  key PK, value          -- vacancy_text, manager_contact, max_invites(=3), тексты пушей
```

`invite_count` = `COUNT(*) FROM attendance WHERE user_id=?`; при `>= max_invites` → `exhausted`. Триггер ищет ближайший прошедший `scheduled` созвон и сверяет `trigger_word`. `call_config` заменяется на `calls` + `attendance`.

## 9. Технические / нефункциональные требования

- Стек: aiogram 3, aiosqlite, APScheduler, systemd. Деплой — `deploy.sh`.
- Объём десятки–сотни/сутки → **SQLite хватает**.
- **Троттлинг рассылки**: батчи + `asyncio.sleep`, обработка `TelegramRetryAfter`. В текущем `_send_ping` нет — добавить.
- **Джобы пушей** привязаны к `call_id` (не к фиксированным id).
- **Перезапуск-устойчивость**: при старте перепланировать пуши по всем будущим `scheduled` созвонам.
- **/start от дублей**: не плодить юзеров, не трать source.
- **Безопасность**: токен и admin_ids только в `.env` (убрать дефолт из `config.py`, токен отозвать).

## 10. Антифрод / риски

- **Утечка триггера** — главный риск. Простой `+` сольётся: любой случайный юзер бота получит контакт. Меры: (A) неочевидное слово на созвон; (B) гейт «только `confirmed`»; (C) окно активности по времени. Дефолт — A+C, B опционально.
- **Качество лида**: триггер-слово само по себе фильтрует до заинтересованных → менеджер не тонет в пустых диалогах.
- **Шейв/фейк-лиды**: source-трекинг + воронка отсекают источники с дешёвым `entered`, но нулевым `contact_issued`.
- **Менеджер**: один аккаунт на поток может словить флуд-репорты → ускорить ротацию (v2).
- **Анти-надоедание**: лимит 3 инвайтов, один инвайт на созвон.

## 11. Этапность

**MVP**
Многосозвонная модель, /start + source, инвайт + общая вакансия, RSVP, пуши (24ч/1ч/10м со ссылкой), триггер-слово → выдача контакта, автореинвайт `declined`/молчунов с лимитом 3, статистика воронки + по source, троттлинг.

**v2**
Ротация менеджеров, гейт триггера по `confirmed`, экспорт/дашборд по источникам, A/B текстов.

**Не входит**
Автогенерация ссылки на созвон, веб-админка, индивидуальные слоты, сбор телефона/доп. данных.

## 12. Зафиксированные решения

1. Присутствие/интерес фиксируется триггер-словом после созвона.
2. Вакансия — один общий текст.
3. Менеджер — один, ротация позже.
4. Лимит приглашений — 3 суммарно (настраивается), один инвайт на созвон.
5. Объём — десятки–сотни/сутки → SQLite.
6. Ссылка на созвон — руками.
7. Доп. данные не собираем, только TG.
8. Время — МСК для всех.
9. Контакт выдаётся не автопушем, а по триггер-слову, названному на созвоне.
